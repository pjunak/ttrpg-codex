## Internationalization (i18n)

UI strings are authored as **English source keys**; translations are
served per-user via a language dropdown on the dashboard. Code/admin
stays English; the player-facing UI renders in the viewer's language.

**Per-user, NOT campaign-wide.** Language is a personal browser choice
(`localStorage['codex_lang']`), never synced to the server like the
theme is. DM and every player (even anonymous) pick independently — the
deliberate divergence from the otherwise-mirrored theme system.

**Resolution:** explicit stored choice → `navigator.languages` primary-
subtag match (`cs-CZ` → `cs`) → English fallback. Loaded once at boot
**before the first render** (`await I18n.load()` in app.js
`DOMContentLoaded`, before the first `Sidebar.render()`), so there's no
flash of the wrong language.

### `web/js/i18n.js` (`I18n`)
Self-contained IIFE — imports NOTHING app-side, touches no DOM at load
(so Node tests import it cleanly). `t()` returns PLAIN text — **callers
`esc()` before innerHTML**, exactly as for plain literals. Public API:
- `load()` — detect locale, set `<html lang>`, fetch en + active catalog
  (idempotent; never refetched on SSE/re-render — only `setLocale` to a
  new language fetches).
- `t(key, params?)` — fallback chain active → en → key; `{placeholder}`
  interpolation; never throws (missing key → the key, or `⟦key⟧` when
  `localStorage['codex_i18n_debug']==='1'`).
- `plural(key, n, params?)` — native `Intl.PluralRules` picks the bucket
  from a `{one,few,many,other}` catalog entry. Czech integers →
  one(1)/few(2-4)/other(0,5+); `n` auto-supplied to interpolation. The
  hand-rolled `czPlural` (constants.js) is **gone**.
- `formatDate(value, opts)` / `formatNumber(n, opts)` — `Intl.*`, active
  locale. Replaced the 3 hardcoded `toLocaleString('cs-CZ')` sites
  (exportJSON, draft banner, snapshot list).
- `relativeTime(ms, now?)` — `Intl.RelativeTimeFormat`; backs the
  `utils.humanTime` shim (now a thin delegate — its many callers are
  unchanged).
- `setLocale(id)` — persist + switch + re-render (async: fetches a not-
  yet-loaded catalog once). Wired to the dropdown.
- `detectLocale(languages?, stored?)` — pure (DOM-free, testable).
- `getLocale()` / `availableLocales()` (`[{id, endonym}]`) /
  `applyLocale()` (sets `<html lang>`) / `hydrate(root)` /
  `register(id, catalog)` (tests + pre-load) / `setRerender(fn)` (app.js
  injects the re-render closure — same late-binding pattern as
  `setWikiLinkResolver`).

### Catalogs — `web/i18n/en.json` (source) + `cs.json`
Flat, dotted keys (`nav.characters`, `cloudmap.events`, …); plural
entries are the only nested values (CLDR-bucket objects). Served by
`express.static(WEB_DIR)` at `/i18n/*.json`, shipped via `COPY web` (no
server/Dockerfile change). Adding a language = one `/i18n/<id>.json` +
one row in `I18n` `LOCALES`.

### Wiring (app.js)
- `I18n` is in the `ACTIONS` map; the dropdown `<select>` dispatches
  `I18n.setLocale($value)` via `dataOn('change', …)`.
- Dashboard chip `_renderLangChip()` mirrors `_renderTopbarLogin`
  (`.topbar-lang` in edit.css; `body.has-lang-chip` stacks the anonymous
  login chip below it). Shown for ALL roles on `/` (route-gated).
- Switch re-render = `_fullChromeRerender()` (the SSE `_applyRemoteChange`
  fan-out **minus** `Store.load`). **Mid-edit guard:** if
  `EditMode.isDirty()`, only the editor-free chrome flips + a toast; the
  full `navigate()` re-render defers to the `editmode:clean` drain
  (`_pendingLangRerender`) so a live EasyMDE/CodeMirror editor isn't
  destroyed with its unsaved text.
- Static `index.html` chrome (loading text, bottom-nav, map sheet,
  search/settings labels) is English by default + carries `data-i18n` /
  `data-i18n-title`; `I18n.hydrate(document)` swaps it at boot + on
  switch.
- Sidebar page labels come from `SIDEBAR_PAGES[].key` via `I18n.t`
  (`Sidebar._label`).

### Scope boundary (chrome vs content)
Only UI **chrome** is translated. User-authored content (names,
descriptions, markdown bodies) AND **DM-editable enum/section labels**
(attitudes, pin types, statuses, relationship types, and sidebar
SECTION labels in `SIDEBAR_LAYOUT_DEFAULT`) are CONTENT — left as
stored. `SIDEBAR_PAGES` link labels (a pure hardcoded registry, never
persisted) ARE translated.

### Status — UI migration COMPLETE (~1000 keys, en + cs)
Every user-facing UI string across the app is now translated: the nav
frame, dashboard, **all** edit forms (`edit_templates.js`/`editmode.js`),
wiki articles + lists, map / timeline / cloudmap chrome, settings (every
tab), search, sidebar editor, widgets, the Addon Manager UI, and the
app.js / store.js banners + error strings. Keys are namespaced per area
(`nav.*`, `action.*` shared verbs, `settings.*`, `editform.*`,
`editmode.*`, `wiki.*`, `map.*`, `cloudmap.*`, `timeline.*`, `dm.*`,
`search.*`, `sidebar.*`, `widget.*`, `addons.*`, `app.*`, `store.*`,
`time.*`, `loading.*`, `mapSheet.*`, `lang.*`).

**Two test guards** (run in `npm test`): `test/i18n.test.mjs` (engine +
catalog parity — cs covers every en key) and **`test/i18n-keys.test.mjs`**
(scans the sources and asserts every LITERAL `I18n.t('…')`/`I18n.plural('…')`
key + every `data-i18n` attribute actually exists in `en.json`). When you
add a new UI string: wrap it in `I18n.t`/`plural` and add the key to BOTH
`web/i18n/en.json` and `cs.json` (the parity + key tests enforce both).

**Deliberately NOT translated (left Czech, by design):** DM-editable
enum/content defaults in `data.js` (relationship types, pin
types, statuses, priorities, attitudes — these are CONTENT, **now seeded
in English** so a fresh first-deploy starts English; existing instances
keep their persisted data) and the enum *id* values used as keys (e.g. the
`'střední'` priority id — kept as the original Czech word: it's persisted
on events + hardcoded in `wiki.js`/`cloudmap.js`, so only its label is
English); the `SIDEBAR_LAYOUT_DEFAULT` section labels + `THEMES` label
(content); the `localeCompare('cs')` sort sites; and **addon-framework
internal diagnostics** (`addon-deps.js` blocked-reason strings,
`addon-fragments.js` conflict message, the thrown `Error(...)` API-violation
messages in `addons.js` — developer-facing, surface only for a misbehaving
installed addon). Third-party addon-rendered UI is also out of scope —
addons are written in **English** (the language switcher is a visual layer
over the *core* UI only; it doesn't reach addon code, and there is no addon
translation API). ⚠ **When migrating a placeholder/
title, check nothing queries the DOM by that text** — a `querySelector(
'input[placeholder="…"]')` breaks once the placeholder is localized (this
bit the rank-chain name input: it now carries a stable `.rank-chain-name`
class that `editmode.saveFaction` queries instead).
